# Operator Controls & Unique Position Identity — v1 Proposal

**Status:** Draft proposal for review  
**Author intent:** Written for Codex, Claude, Gemini, and the human operator to audit before implementation.  
**Primary goal:** Give the operator safe, precise, auditable control over live bot risk without turning the bot into a manual trading terminal.

---

## 1. Problem

The bot has strong automated controls: strategy gates, risk sizing, sleeve allocation, reconciliation, kill switches, and execution logging. What it does not yet have is a clean operator command layer.

Today, if the operator wants to intervene, the choices are too blunt:

- Let the strategy exit on its normal mechanics.
- Manually close something in Alpaca.
- Stop or restart the bot.
- Edit config/code and recycle the process.

That is not good enough for live trading.

The operator needs to be able to:

- List currently open bot-managed positions.
- Identify each position unambiguously.
- Close or partially reduce one exact position.
- Pause new entries without disabling existing position management.
- Halt the bot when something is wrong.
- Review what commands were issued and what the bot actually did.

The most important design requirement is unique position identity. Every new logical position must receive a globally unique, immutable ID. Operator commands should target that ID, not ambiguous labels like "close SMA NVDA."

---

## 2. Core Invariant

Every logical position lifecycle gets one unique, immutable `position_uid`.

This ID:

- Is generated by the bot when a new position lifecycle begins.
- Is never reused, even after the position is closed.
- Is persisted in SQLite.
- Survives bot restarts.
- Is attached to every related trade, order, operator command, alert, and dashboard/state snapshot.
- Is the primary target for destructive operator commands.

Example:

```text
pos_f3b1a6d9e9554d2093c47f03c46ef9c1
```

Recommended generation:

```python
import uuid

position_uid = f"pos_{uuid.uuid4().hex}"
```

Rationale:

- `uuid4` is built into Python.
- It does not require a new dependency.
- It is safe offline.
- It avoids deriving identity from mutable position metadata.
- It gives practical uniqueness across open, closed, old, and future positions.

Do not call this a "hash" in the schema unless it is actually derived from content. The safer name is `position_uid` or `position_instance_id`.

---

## 3. Identity Model

Alpaca positions are broker-level aggregates. Orders have IDs; open positions are addressed by symbol or asset id. Alpaca does not provide a durable "position instance ID" for each lifecycle of a position.

Relevant Alpaca concepts:

- Orders have `id` and `client_order_id`.
- Open positions are closed by `symbol_or_asset_id`.
- All open positions can be liquidated through the positions API.
- All open orders can be canceled through the orders API.
- Account configuration can block new trading via `suspend_trade`.

References:

- Alpaca close position: https://docs.alpaca.markets/us/reference/deleteopenposition-1
- Alpaca close all positions: https://docs.alpaca.markets/us/reference/deleteallopenpositions-1
- Alpaca cancel all orders: https://docs.alpaca.markets/us/reference/deleteallorders-1
- Alpaca account configuration: https://docs.alpaca.markets/us/reference/patchaccountconfig-1
- Alpaca working with orders: https://docs.alpaca.markets/docs/working-with-orders

The bot should therefore separate broker identity from bot identity:

| Concept | Meaning | Example |
|---|---|---|
| `broker_symbol` | Alpaca aggregate position key | `NVDA` |
| `owner_key` / current engine `position_id` | Bot's current management key for broker aggregation | `NVDA`, `SPY`, or spread UUID |
| `position_uid` | Unique lifecycle ID for this specific opened position | `pos_f3b1...` |
| `order_id` | Alpaca order ID | broker generated |
| `client_order_id` | Bot-provided order ID, should include or map to `position_uid` | `bot_pos_f3b1_reduce_...` |

The current engine already has `engine.positions.Position.position_id`. For single-leg equities this is currently the symbol, which is useful for live ownership but is not enough for lifecycle identity.

Proposed rule:

```text
Keep existing position_id / owner_key behavior for broker aggregation.
Add position_uid for immutable lifecycle identity.
```

## 4. Operator Experience

The operator should list open positions first:

```bash
./operator.py positions
```

Example output:

```text
POSITION_UID                              SYMBOL  STRATEGY       QTY   ENTRY    MARKET   UPNL    AGE   STATUS
pos_f3b1a6d9e9554d2093c47f03c46ef9c1     NVDA    sma_crossover  12    884.20   901.10   +202.80  6d    open
pos_91cfaae6f08f43169bb4a4f7b33d87ae     MU      sma_crossover  34    116.30   121.00   +159.80  4d    open
```

Then target one exact position:

```bash
./operator.py reduce-position pos_f3b1a6d9e9554d2093c47f03c46ef9c1 --pct 50 --reason "take profit before event risk"
./operator.py close-position pos_f3b1a6d9e9554d2093c47f03c46ef9c1 --reason "manual exit before SMA signal"
```

The CLI should not directly place broker orders. It should write a command request. The running bot validates and executes the command inside the normal engine context.

```text
operator.py -> command queue -> engine validates -> broker action -> audit result
```

Rationale:

- Keeps all broker mutations inside the engine.
- Reuses ownership, stop, sleeve, reconciliation, and logging logic.
- Prevents an external CLI from bypassing safety checks.
- Gives the bot a chance to reject stale, invalid, or unsafe commands.

## 5. V1 Command Surface

V1 should be small, risk-reducing, and auditable.

### 5.1 Read-only commands

| Command | Example | Rationale |
|---|---|---|
| `status` | `./operator.py status` | Show whether the bot is running, latest cycle time, startup mode, stream health, active controls, open positions, and open orders. First diagnostic command. |
| `positions` | `./operator.py positions` | List all open bot-managed positions with `position_uid`, symbol, strategy, qty, entry, market value, unrealized P&L, age, stop/order state. Required before targeted actions. |
| `show-position` | `./operator.py show-position pos_...` | Show full metadata for one position: entry order, fills, current broker qty, stop/order state, strategy owner, realized/unrealized P&L, and prior operator actions. |
| `commands` | `./operator.py commands` | Show recent operator commands and their status: pending, accepted, rejected, executing, succeeded, failed. Essential for audit and debugging. |

### 5.2 Soft controls

| Command | Example | Rationale |
|---|---|---|
| `pause-entries` | `./operator.py pause-entries --reason "market event"` | Stop new positions while allowing existing positions to remain managed. Safest first intervention when the operator is uncertain. |
| `resume-entries` | `./operator.py resume-entries --reason "event passed"` | Re-enable new entries after reconciliation/preflight. Should not clear a hard halt. |
| `pause-strategy` | `./operator.py pause-strategy sma_crossover --reason "SMA regime concern"` | Stop new entries for one strategy only. Useful when one strategy looks wrong but the rest of the bot can continue. |
| `resume-strategy` | `./operator.py resume-strategy sma_crossover --reason "review complete"` | Restore one paused strategy after checks. Keeps intervention scoped. |

### 5.3 Position controls

| Command | Example | Rationale |
|---|---|---|
| `reduce-position` | `./operator.py reduce-position pos_... --pct 50 --reason "take profit"` | Partial close by exact lifecycle ID. Critical for profit-taking, event-risk reduction, and operator risk management. |
| `close-position` | `./operator.py close-position pos_... --reason "manual exit"` | Full close by exact lifecycle ID. Safer than symbol/strategy labels. |
| `cancel-position-orders` | `./operator.py cancel-position-orders pos_... --reason "stale protective order"` | Cancel open orders tied to one position. Useful before a reduce/close or when an order is stale. |

### 5.4 Emergency control

| Command | Example | Rationale |
|---|---|---|
| `halt` | `./operator.py halt --reason "unexpected fills"` | Sticky emergency stop. Blocks new entries and requires explicit manual recovery. Should require confirmation. |
| `resume-after-halt` | `./operator.py resume-after-halt --reason "reconciled"` | Clear a hard halt only after broker reconciliation, stream health check, DB ownership check, and preflight. Intentionally harder than `resume-entries`. |

V1 minimum viable subset:

```bash
./operator.py status
./operator.py positions
./operator.py show-position <position_uid>
./operator.py pause-entries --reason "..."
./operator.py resume-entries --reason "..."
./operator.py reduce-position <position_uid> --pct 50 --reason "..."
./operator.py close-position <position_uid> --reason "..."
./operator.py halt --reason "..."
./operator.py commands
```

## 6. Authentication & Authorization

For v1, local OS-level access is acceptable as the primary authentication boundary.

Assumption:

If an operator can SSH into the VPS as the bot operator user, run the bot virtualenv, read the bot config, and access the trade DB, they already have sufficient operational authority to affect the bot.

However, v1 should still enforce layered safety:

- The operator command queue must be writable only by the bot OS user.
- Destructive commands require `--reason`.
- Destructive commands should require explicit confirmation, such as `--confirm <position_uid>` or a command-specific confirmation string.
- All commands must be audited.
- The bot validates commands before execution.
- The CLI is local-only; no public web write controls in v1.

This avoids building a web authentication system too early while still giving accountability and accident prevention.

If a future dashboard adds write controls, that becomes a separate security project:

- Login
- MFA
- CSRF protection
- Session expiry
- Role-based permissions
- Network allowlisting
- Command signing or durable audit identity

## 7. Command Queue & Audit Trail

The CLI should write command requests to SQLite or another durable local queue. SQLite is preferred because the project already uses it for trade logging.

Suggested table: `operator_commands`

| Column | Type | Notes |
|---|---|---|
| `id` | INTEGER PRIMARY KEY | Local DB row id |
| `command_uid` | TEXT UNIQUE NOT NULL | `cmd_<uuid4_hex>` |
| `created_at` | TEXT NOT NULL | UTC ISO timestamp |
| `requested_by` | TEXT | OS user or configured operator name |
| `action` | TEXT NOT NULL | `reduce_position`, `close_position`, `pause_entries`, etc. |
| `target_position_uid` | TEXT | Required for position-specific actions |
| `target_strategy` | TEXT | Required for strategy-specific controls |
| `params_json` | TEXT NOT NULL | Qty, pct, flags, confirmation, etc. |
| `reason` | TEXT NOT NULL | Required for state-changing commands |
| `status` | TEXT NOT NULL | `pending`, `accepted`, `rejected`, `executing`, `succeeded`, `failed` |
| `result_json` | TEXT | Broker result, rejection reason, final state |
| `accepted_at` | TEXT | Bot accepted timestamp |
| `completed_at` | TEXT | Bot completed timestamp |

Commands should be append-only from the operator's perspective. The bot updates status/result fields as it processes the command.

Important: The command audit trail should not contain secrets, raw account balances, or sensitive broker payloads. Store high-level state and IDs.

### 7.1 Crash recovery and idempotency

The command queue must be designed so a bot crash cannot double-dispatch an operator order.

Required rule:

```text
Before sending any network request to Alpaca, the engine must durably mark the
command as executing and persist the intended client_order_id.
```

Recommended processing sequence:

1. Engine reads one `pending` command.
2. Engine validates the command.
3. Engine generates deterministic command-scoped `client_order_id`.
4. Engine writes `status='executing'`, `accepted_at`, and `client_order_id`.
5. Engine commits the SQLite transaction.
6. Engine sends the broker request.
7. Engine records success/failure and final broker/order details.

On restart:

- Stale `pending` commands older than a short expiration window should be rejected, not executed.
- `executing` commands with a `client_order_id` should be reconciled against Alpaca orders before any retry.
- A command with an existing `client_order_id` must never dispatch a second order until reconciliation proves no broker-side order exists.
- If reconciliation cannot prove whether a destructive command reached Alpaca, mark the command `indeterminate` / `needs_operator_review` instead of automatically retrying.

Suggested v1 expiration:

```text
pending command older than 60 seconds at engine pickup or startup => rejected_expired
```

This is intentionally conservative. An operator can always re-issue the command after checking current state.

Implementation requirement:

- `OPERATOR_COMMAND_EXPIRY_SECONDS = 60` should live in `config/settings.py`.
- Operator command polling must run on a fast heartbeat independent of slow strategy cycles; otherwise use a longer expiry such as 180-300 seconds.
- `client_order_id` strings must fit Alpaca's documented client order ID constraints. Keep generated IDs short and deterministic, for example `op_<command_uid_short>_<action_short>`, with the full mapping stored in SQLite.

## 8. Trade Log Schema Changes

The current `trades` table already has `position_id` and `position_type`.

Proposed additions:

| Column | Type | Rationale |
|---|---|---|
| `position_uid` | TEXT | Immutable lifecycle identity. |
| `parent_position_uid` | TEXT nullable | Future optional field for derived positions or rolls. Not required for v1 if too speculative. |
| `operator_command_uid` | TEXT nullable | Link trade/fill rows to the operator command that caused them. |
| `client_order_id` | TEXT nullable | Store bot-provided client order ID separately from Alpaca `order_id`. |

Suggested indexes:

```sql
CREATE UNIQUE INDEX IF NOT EXISTS idx_position_uid_unique_lifecycle
ON position_lifecycle(position_uid);

CREATE INDEX IF NOT EXISTS idx_trades_position_uid
ON trades(position_uid);

CREATE INDEX IF NOT EXISTS idx_operator_commands_position_uid
ON operator_commands(target_position_uid);
```

Potential separate table: `position_lifecycle`

| Column | Type | Notes |
|---|---|---|
| `position_uid` | TEXT PRIMARY KEY | Immutable ID |
| `created_at` | TEXT NOT NULL | Lifecycle creation timestamp |
| `closed_at` | TEXT nullable | When fully closed |
| `symbol` | TEXT NOT NULL | Broker symbol at open |
| `owner_key` | TEXT NOT NULL | Current engine/broker aggregation key |
| `strategy` | TEXT NOT NULL | Owning strategy |
| `position_type` | TEXT NOT NULL | `single_leg`, `spread`, etc. |
| `status` | TEXT NOT NULL | `pending`, `open`, `partially_filled`, `reducing`, `closing`, `closed`, `canceled`, `external_closed`, `error` |
| `entry_qty` | REAL | Intended or final entry quantity |
| `current_qty` | REAL | Remaining open quantity |
| `avg_entry_price` | REAL | VWAP of entry fills |
| `net_realized_pnl` | REAL | Running realized P&L for partial/full exits |
| `entry_order_id` | TEXT nullable | Alpaca opening order id |
| `entry_client_order_id` | TEXT nullable | Bot opening client order id |
| `first_fill_at` | TEXT nullable | First entry fill timestamp |
| `last_fill_at` | TEXT nullable | Most recent fill timestamp |
| `metadata_json` | TEXT | Non-sensitive lifecycle metadata |

This table gives the operator a stable source for `positions`, `show-position`, and history.

For single-leg positions and spreads, store legs in a normalized child table:

```sql
CREATE TABLE IF NOT EXISTS position_lifecycle_legs (
    id                  INTEGER PRIMARY KEY AUTOINCREMENT,
    position_uid        TEXT NOT NULL,
    symbol              TEXT NOT NULL,
    side                TEXT NOT NULL,
    qty                 REAL NOT NULL,
    avg_entry_price     REAL,
    FOREIGN KEY(position_uid) REFERENCES position_lifecycle(position_uid)
);
```

Rationale: this mirrors `engine.positions.Position` / `PositionLeg`, avoids stuffing spread legs into JSON/string blobs, and makes options/spread restart reconstruction auditable.

### 8.1 Lifecycle creation timing

`position_uid` should be generated before broker submission, at order planning/building time.

Rationale:

- The bot can embed or map it into `client_order_id`.
- Async fills and REST order lookups can immediately map back to the intended lifecycle.
- Entry intents that are accepted, partially filled, canceled, or rejected still have a durable audit identity.

Lifecycle state recommendation:

1. Create `position_lifecycle` with `status='pending'` before dispatch.
2. First fill transitions to `open` or `partially_filled`.
3. Additional entry fills update `current_qty` and `avg_entry_price`.
4. Canceled entry with zero fills transitions to `canceled`.
5. Canceled entry with partial fills remains open at the filled quantity.

`created_at` means lifecycle creation/planning time, not first fill time. Fill timestamps belong in `trades`, `first_fill_at`, `last_fill_at`, or leg rows.

## 9. Bot-side Validation Rules

The bot must validate every state-changing command before acting.

For `reduce-position`:

1. `position_uid` exists.
2. Position lifecycle status is open or reducible.
3. Broker still has the corresponding symbol/legs.
4. The strategy recorded in DB still owns that position.
5. Requested `qty` or `pct` is valid.
6. Requested reduce quantity does not exceed broker quantity.
7. Existing close orders are checked to avoid duplicate exits.
8. Protective stop order quantity is canceled/replaced or otherwise reconciled.
9. The order generated is risk-reducing only.
10. Result is logged back to `operator_commands` and `trades`.

For `close-position`:

1. Same checks as reduce.
2. Close quantity is current broker quantity for that position.
3. Any pending close/reduce order blocks duplicate close unless operator passes a specific override flag.
4. Position remains tracked until the broker confirms fill/close.
5. Lifecycle status changes to closed only after confirmation.

For `pause-entries`:

1. Set global control state to block new entries.
2. Existing position management continues.
3. State is persisted.
4. Alert is sent.

For `halt`:

1. Set sticky `HALTED` state.
2. Block new entries.
3. Keep existing broker-side protective stops active.
4. Do not cancel existing protective stops as part of normal `halt`.
5. Optionally expose Alpaca `suspend_trade=true` later as a separate hard-broker-lock command, not as the default v1 halt behavior.
6. Require `resume-after-halt` with reconciliation before new entries can resume.

Rationale: a bot halt should stop the bot from increasing or reshaping risk, but canceling broker-resident protective stops can leave existing positions unprotected. Account-level `suspend_trade=true` may also block risk-reducing orders and should be reserved for a separate, heavily confirmed emergency command.

## 10. Partial Close Semantics

Partial closes must be handled as first-class lifecycle events.

Example:

```bash
./operator.py reduce-position pos_f3b1... --pct 50 --reason "take profit"
```

Expected behavior:

1. Bot validates the command.
2. Bot computes the reduce quantity from current broker quantity.
3. Bot cancels or adjusts any pending protective order if needed.
4. Bot submits a reducing order.
5. Bot records the order with `position_uid`.
6. Bot records fill(s) as partial close rows.
7. Bot leaves the position open if remaining qty > 0.
8. Bot updates entry/risk context as needed for remaining quantity.
9. Bot keeps the same `position_uid` for the remaining position.
10. Bot marks the lifecycle closed only when remaining broker quantity reaches zero.

Open design question:

Should a partial close reduce the original initial risk basis, leave it unchanged for R-multiple reporting, or record a separate realized R for the reduced portion while retaining the original basis for the remaining quantity?

Recommendation for v1:

Record each partial close as a realized event linked to the same `position_uid`. Keep lifecycle-level reporting simple:

- realized P&L so far
- remaining open qty
- unrealized P&L on remaining qty
- total lifecycle P&L once fully closed

For R-multiple reporting, calculate each partial close event against the initial risk allocated to the closed portion:

```text
event_r = realized_pnl / (initial_risk_per_share * closed_qty * multiplier)
```

The lifecycle-level final R can then be computed from cumulative realized P&L against the total initial risk once the position is fully closed.

Health/edge reporting must be reviewed so partial exits do not accidentally inflate trade count unless "trade count" is intentionally defined as fill/exit-event count. Lifecycle-level reporting should be able to distinguish one position with three partial exits from three independent positions.

## 11. Stop/Order Interaction

This is one of the highest-risk implementation areas.

If a position has an active protective stop and the operator reduces the position, the bot must not leave an oversized stop that can flip the account short or close too much.

For v1, before a manual reduce/close:

- Find open orders linked to `position_uid`, symbol, strategy, or current close-side orders.
- If there is a pending close order, reject the command unless the operator explicitly cancels it first.
- If there is a protective stop, cancel it before the reduce/close and wait for cancellation confirmation through stream or REST.
- After a partial reduce fill, submit a new protective stop for the remaining quantity if strategy/risk rules require it.
- After a full close fill, do not re-create a stop.

This should be tested carefully.

Recommended sequence:

1. Lock the symbol/position lifecycle as `reducing` or `closing`.
2. Freeze strategy decisions for that symbol while the operator command is active.
3. Cancel the existing stop and wait for broker confirmation.
4. Submit the reduce/close order.
5. After fill, recreate a correctly sized stop if quantity remains.
6. Release the lock after broker and local state reconcile.

Known race: a broker-side stop can fill while the bot is trying to cancel it. The implementation must treat this as a first-class reconciliation path: if the stop fills, the operator command should be rejected or marked superseded rather than submitting an additional reduce/close.

V1 order-type default:

- For equity/manual SMA-style closes and reductions, default to market or marketable orders after stop cancellation confirmation to minimize the unprotected window.
- For options and spreads, do not default to plain market orders. Use strategy-specific close logic or marketable limits because bid/ask behavior can be much worse.

### 11.1 Symbol-level locks

Because Alpaca aggregates positions by symbol, the engine must serialize state-changing actions at the symbol/owner-key level.

While a position is `opening`, `reducing`, or `closing`:

- No other operator command may mutate the same symbol/owner key.
- No strategy may submit a new entry for the same symbol/owner key.
- No engine-dispatched strategy exit should compete with the operator command unless it explicitly reconciles the active lock and supersedes the operator command.

This is required even with unique `position_uid`; broker reality is still symbol-aggregated for equities.

Broker-resident protective stops remain active and can fill naturally. If that happens while an operator command is pending/executing, the engine must mark the operator command `superseded` or `indeterminate` based on reconciliation, rather than sending an additional close.

Central lock registry requirement:

- The engine should maintain one registry for symbol/owner-key locks.
- Every broker mutation path must check the registry immediately before dispatch: normal equity orders, single-leg option workers, spread dispatch, close paths, stale-order cleanup, recovery paths, and operator commands.
- The lock check should happen as close as possible to the actual broker API call, not only at signal-generation time.

## 12. V1 Non-goals

V1 should not include:

- Opening new arbitrary positions from operator commands.
- Manual strategy signal injection.
- Public web dashboard write controls.
- Multi-user role management.
- Automatic reallocation based on operator commands.
- Complex roll workflows.
- Tax-lot selection.
- Cross-account support.
- Direct CLI broker mutations that bypass the engine.

These can be considered later, but they are not needed to solve the safety problem.

## 13. Implementation Sketch

Possible implementation sequence:

### Phase A: read-only identity

1. Add `position_uid` generation before order submission.
2. Add `position_lifecycle` table and migrations.
3. Add `position_lifecycle_legs` and write single-leg/spread leg rows.
4. Attach `position_uid` to trade log rows and state snapshot.
5. Add read-only `operator.py status`, `positions`, `show-position`, `commands`.
6. Verify restart rehydrates lifecycle IDs correctly.

No broker-mutating operator commands should ship in Phase A. This lets reviewers validate the identity model before any operator command can trade.

### Phase B: soft and emergency controls

1. Add `operator_commands` queue table.
2. Add engine polling/processing of pending operator commands.
3. Add fast command heartbeat, crash-safe command state transitions, command expiration, and `indeterminate` reconciliation state.
4. Implement `pause-entries` and `resume-entries`.
5. Implement `pause-strategy` and `resume-strategy`.
6. Implement `halt` and `resume-after-halt`.
7. Add alerts for all command state transitions.

### Phase C: destructive position controls

1. Implement symbol/owner-key locks.
2. Implement `cancel-position-orders`.
3. Implement `close-position`.
4. Implement `reduce-position`.
5. Implement stop cancel/recreate handling.
6. Wire manual close/reduce fills into lifecycle, trade log, PnL, and sleeve allocator state.
7. Verify in dry-run and paper trading before any live use.

## 14. Testing Requirements

Unit tests should cover:

- Every new position gets a unique `position_uid`.
- Closed IDs are never reused.
- Same symbol reopened later gets a new `position_uid`.
- Restart restores open position lifecycle IDs from SQLite.
- Operator cannot act on a closed `position_uid`.
- Operator cannot act on an unknown `position_uid`.
- Operator cannot reduce more than current broker qty.
- Operator command is rejected if ownership is ambiguous.
- Operator command is rejected if a close order is already pending.
- Partial reduce keeps lifecycle open.
- Full close marks lifecycle closed only after fill confirmation.
- Stop quantity is not left oversized after partial reduce.
- Stop-fill-during-cancel race is reconciled without double-closing.
- Command marked `executing` before broker dispatch is not double-dispatched after restart.
- Stale pending command expires without execution.
- `executing` command with `client_order_id` reconciles on restart.
- Destructive command with ambiguous broker reconciliation becomes `indeterminate` / `needs_operator_review`, not auto-retried.
- Generated `client_order_id` fits Alpaca constraints and maps back to the full command/position IDs.
- Manual close/reduce updates sleeve allocator realized P&L and releases used capital.
- Direct external close maps to the open lifecycle and marks `external_closed`.
- Partial exits do not distort lifecycle-level trade counts in health/edge reports.
- `position_lifecycle_legs` restores single-leg and spread positions correctly.
- Symbol-level lock blocks competing strategy/operator mutation.
- Symbol-level lock is checked immediately before every broker dispatch path, including async option/spread workers.
- Broker-side stop fill during operator command marks the operator command superseded/indeterminate rather than double-closing.
- Halt blocks entries.
- Halt does not cancel existing broker-side protective stops.
- Resume after halt requires reconciliation/preflight.

Integration/manual paper checks should cover:

- List positions while bot is running.
- Partially reduce a paper SMA position.
- Fully close a paper SMA position.
- Verify trade log, lifecycle table, command table, alerts, and Alpaca state agree.
- Restart bot and verify closed/open lifecycle state remains correct.

## 15. Review Questions

Claude/Gemini reviewers should specifically audit the remaining unresolved or high-risk choices:

1. Is the proposed crash-recovery sequence sufficient to prevent double dispatch?
2. Is 60 seconds the right expiration window if command processing runs on a fast heartbeat, or should it be longer?
3. Is the stop cancel/recreate sequence safe enough for fast markets with equity market/marketable exits?
4. Which strategy-driven exits, if any, are allowed to supersede an operator lock, and how should that be audited?
5. Does `position_lifecycle_legs` cover every options/spread restart case already present in the engine?
6. Do `first_fill_at` / `last_fill_at` plus trade rows provide enough lifecycle timing detail?
7. Should short-prefix matching be allowed for destructive commands if interactive confirmation is present, or should v1 require full IDs only?
8. Should Alpaca `suspend_trade=true` stay fully deferred, or should the doc mention it only as an out-of-band Alpaca dashboard action?
9. Are there lifecycle/PnL edge cases for partial closes that would distort existing health/edge reports?
10. Are there places in the current engine where symbol-level locks could be bypassed by async options/spread workers?

## 16. Recommended Decisions

Recommended defaults for v1:

- Use `position_uid = "pos_" + uuid.uuid4().hex`.
- Store `position_uid` in both a lifecycle table and each trade row.
- Generate `position_uid` before broker submission.
- Create lifecycle records in `pending` state before dispatch.
- Mark operator commands `executing` and persist `client_order_id` before broker dispatch.
- Expire stale pending commands rather than executing old intent.
- Use a fast operator-command heartbeat if `OPERATOR_COMMAND_EXPIRY_SECONDS = 60`; otherwise choose a longer expiry.
- Mark ambiguous destructive-command reconciliation as `indeterminate` / `needs_operator_review`; do not auto-retry.

- Have the CLI write a command request, not call Alpaca directly.
- Let the engine validate and execute commands.
- Keep existing broker-side protective stops active during `halt`.
- Defer Alpaca `suspend_trade=true` from v1.
- Serialize state-changing activity with symbol/owner-key locks.
- Require every broker dispatch path to check the central lock registry immediately before the API call.
- Default equity manual closes/reductions to market or marketable orders after stop cancellation confirmation; keep options/spreads on strategy-specific or marketable-limit close logic.
- Wire manual close/reduce fills into trade logging, lifecycle state, PnL, and sleeve allocator updates.
- Start with read-only commands and identity persistence before adding broker-mutating controls.

The main principle: operator controls should reduce ambiguity and reduce risk. They should not create a second uncontrolled trading path.
